Nimbus Parallel Flow

A parallel processing library for Roblox. API reference, version V20.

Overview

The library distributes work across worker Actors, runs it in parallel, and returns the results in order. Actor messaging, binary serialization, scheduling and thread synchronization are handled internally. A worker is a ModuleScript returning a function, applied once per element of the input.

The sequential-versus-parallel decision is made at runtime from a measured cost model: small inputs run sequentially, because the round-trip cost of crossing the actor boundary exceeds the compute; larger inputs are chunked across the pool. 

local ParallelFlow = require(game.ReplicatedStorage.ParallelFlow)

local pool = ParallelFlow.createPool({ size = 8 })
local results = pool:map({ 1, 2, 3, 4 }, MyWorker)

How it works

Data is serialized into a buffer before crossing the actor boundary and decoded on the worker side. Moving a buffer between actors is cheaper for the engine than moving structured tables, so the serializer keeps encode and decode cost low: per-type fast paths, buffer reuse through a re-entrant free-list, and binary paths that skip serialization on the result entirely.

Two runtime measurements drive every map: the per-element cost of the worker, micro-calibrated over a small sample, and the pool's pure round-trip latency, measured with a calibration round trip when the pool starts. If the compute hides under the round-trip floor, the work runs sequentially; otherwise it is chunked across the workers.

Installation

The library ships as a single assembled package. There are no scripts to place individually and no internal structure to assemble.

  1. Obtain the package (the Creator Store model or the .rbxm file).
  2. Insert it into ReplicatedStorage. It contains the serializer, the type codecs and the engine.
  3. Require it where needed:
local ParallelFlow = require(game.ReplicatedStorage.ParallelFlow)

With Rojo or Wally, point the dependency at ReplicatedStorage.ParallelFlow.

The library runs on both the server and the client. size sets how many worker Actors the pool creates, and createPool clamps it to 3 on the client and 64 on the server. How many of those Actors actually run at the same time is decided by the Roblox engine, not the library: it schedules them across a small pool of worker threads whose count depends on the environment and is not guaranteed. In practice this ceiling is well below the Actor cap, especially in a live game. The same config works on both sides; the effective parallelism differs. See Notes and limits.

Quickstart

1. Write a worker

A worker is a ModuleScript that returns a function. The pool calls it once per element. It runs inside an isolated actor and does not share state with the rest of the game while it executes.

-- Square.luau
return function(value)
    return value * value
end

2. Create a pool

Create the pool once and reuse it. Creating a pool spawns the workers and runs a short calibration, so it is not done per frame.

local ParallelFlow = require(game.ReplicatedStorage.ParallelFlow)
local pool = ParallelFlow.createPool({ size = 8 })

3. Map over data

map runs the worker over the list and returns the results in order, plus an errors list (nil when nothing failed). It chooses sequential or parallel execution on its own.

local Square = game.ReplicatedStorage.Workers.Square

local numbers = {}
for i = 1, 5000 do numbers[i] = i end

local results, errors = pool:map(numbers, Square)

print(results[1])     --> 1
print(results[5000])  --> 25000000

4. Many independent jobs

When the work is a set of separate jobs rather than one list, use submit. It returns a Future immediately and coalesces all submissions made in the same frame into roughly one round trip per worker.

local futures = {}
for i = 1, 100 do
    futures[i] = pool:submit(Square, i)
end

local results = pool:awaitAll(futures)

5. Release the pool

pool:destroy()

Pools and workers

A pool owns a set of worker actors and the scheduler that feeds them. It is meant to live across many calls, so create it once and keep it. A worker is a plain ModuleScript returning a function, applied once per element.

Parallelization pays off when the worker performs real per-element computation (physics, geometry, CFrame transforms), which is where the cost of splitting work across threads is offset. For trivial work the round trip dominates, and the adaptive scheduler runs it sequentially.

createPool

ParallelFlow.createPool(config?)

Returns a Pool.

Spawns the workers, waits for each to bind its handlers, and runs a short latency calibration. The config table is optional; every field has a default.

local pool = ParallelFlow.createPool()             -- all defaults
local pool = ParallelFlow.createPool({ size = 8 }) -- override what you need

Configuration

ParallelFlow.createPool({
    size                  = 4,          -- number of workers
    timeout               = 10,         -- seconds before a job is dropped
    adaptive              = true,       -- choose sequential or parallel automatically
    loadBalanceFactor     = 1,          -- chunks per worker (>1 improves balancing)
    errorMode             = "continue", -- "continue" or "strict"
    maxFrameMs            = 6,          -- per-frame dispatch time budget
    maxChunkSize          = nil,        -- max elements per chunk (nil = no cap)
    smallDatasetThreshold = 50,         -- sequential below this when adaptive is off
    sharedTableThreshold  = 1000,       -- use a SharedTable above this size
    nestedDataThreshold   = 10000,      -- SharedTable threshold for nested data
    seedRoundTripMs       = 7,          -- initial round-trip estimate before calibration
    calibrationSamples    = 16,         -- samples used for cost estimation
    onError               = nil,        -- function(message) called on failures
})
OptionDefaultDescription
size4Number of worker Actors the pool creates. Clamped to 3 on the client and 64 on the server. The actual parallelism is lower and set by the engine's worker-thread count, not by this value; see Notes and limits.
timeout10Seconds before a job that has not returned is dropped and counted as timed out.
adaptivetrueWhen true, the sequential-versus-parallel choice comes from the measured cost model. When false, it uses smallDatasetThreshold.
loadBalanceFactor1Chunks emitted per worker. Above 1 produces more, smaller chunks: better balancing under uneven work, at the cost of more round trips.
errorMode"continue""continue" collects per-element errors and returns partial results; "strict" raises on the first failed element.
maxFrameMs6Per-frame time budget for dispatching chunks. The loop yields when it exceeds this, so a large input does not stall a frame.
maxChunkSizenilUpper bound on elements per chunk. nil means no explicit cap.
smallDatasetThreshold50When adaptive is off, inputs smaller than this run sequentially.
sharedTableThreshold1000Inputs at or above this size may be passed through a SharedTable instead of copied per chunk.
nestedDataThreshold10000The SharedTable threshold used when elements are themselves tables.
seedRoundTripMs7Round-trip estimate used before, or if unavailable, the startup calibration.

The two values that most affect behavior are adaptive (whether a given input is worth parallelizing at all) and size (the ceiling on speedup for inputs that are).

Running work

These methods apply a worker across an input and block until it finishes. For the non-blocking forms, see Async and submit.

pool:map(list, module)

Returns results, errors. Blocking.

Runs the worker over the list and returns the results in input order. errors is nil when nothing failed, otherwise a list of { index, message }. Chooses sequential or parallel from the cost model (or smallDatasetThreshold when adaptive is off), and may route large or nested inputs through a SharedTable automatically.

pool:Map(data, module)

Returns results. Blocking.

Router that dispatches by data type: a buffer goes to mapBuffer (stride 8), a list of Vector3 goes to mapVector3, any other list goes to map.

pool:mapTables(list, module)

Returns results, errors. Blocking.

Like map, but always parallel: it skips the sequential decision and the calibration step. Use it when the input is known to be large enough to parallelize.

pool:dispatch(module, data)

Returns the result. Blocking.

Runs the worker on a single value. For more than one value, prefer map (a collection) or submit (independent jobs), which are far cheaper per item.

pool:filter(list, module)

Returns the kept list. Blocking.

Runs the worker as a predicate and keeps the original elements whose result is truthy, preserving order.

pool:Pipeline(data, { modA, modB, modC })

Returns the final result. Blocking.

Runs each module in sequence, feeding the result of one stage into the next through Map. Each stage is parallelized on its own.

Binary and shared paths

These hand the worker raw buffer memory instead of decoded values, avoiding serialization for tightly packed numeric data.

pool:mapVector3(vectors, module)

Returns results. Binary path.

Iterates a list of Vector3. Each vector is packed as three f32 values (12 bytes). The worker receives (buffer, offset):

-- worker
return function(buf, offset)
    local x = buffer.readf32(buf, offset)
    local y = buffer.readf32(buf, offset + 4)
    local z = buffer.readf32(buf, offset + 8)
    return x + y + z
end

pool:mapBuffer(data, stride, module)

Returns results. Binary path.

Binary input, list output. stride is the byte size of one element. The worker receives (buffer, offset) and returns one value per element.

pool:mapBufferOut(data, stride, outStride, module)

Returns a buffer. Binary path.

Binary input and binary output, with no serialization on the result. The worker receives (inBuffer, inOffset, outBuffer, outOffset) and writes directly into the output buffer. This is the lowest-overhead path when both ends are numeric.

-- worker
return function(inBuf, inOff, outBuf, outOff)
    local v = buffer.readf32(inBuf, inOff)
    buffer.writef32(outBuf, outOff, v * 2)
end

pool:mapInto(list, sharedTable, module)

Returns nothing. Shared path.

Writes each result into the SharedTable you pass, at the matching index. The target must be a SharedTable; a plain table is rejected, because writes to it would not survive crossing the actor boundary and would be silently lost.

Async and submit

pool:submit(module, arg)

Returns a Future.

Queues one job and returns a Future immediately. Submissions made within the same frame are buffered and coalesced: instead of one round trip per job, they are split across the workers and sent as roughly one message per worker. A loop of dispatch would instead pay a full round trip each.

local futures = {}
for i = 1, 100 do
    futures[i] = pool:submit(MyJob, i)
end

local results = pool:awaitAll(futures)

pool:awaitAll(futures)

Returns the results list.

Waits for a list of Futures and returns their results in the same order.

Async map variants

Each returns a Future that resolves to the same value its blocking form would return.

local future = pool:mapAsync(list, module)
local future = pool:mapTablesAsync(list, module)
local future = pool:FilterAsync(list, module)
local future = pool:PipelineAsync(data, steps)

local results = future:Await()

Futures

A Future represents a job that has not finished yet. Every async method and submit returns one. Waiting is event-based, not polling: a thread parked in Await or Wait is resumed when the job settles or is cancelled.

MethodDescription
future:Await()Yields until the job settles and returns its result (nil if cancelled or failed).
future:Wait()Yields until the job settles, returns nothing.
future:OnComplete(fn)Registers fn(result, ok) to run once the job finishes. Fires immediately if already settled.
future:Cancel()Cancels the job. A result that arrives after cancellation is ignored.
future:IsCompleted()True once the job has finished.
future:IsCancelled()True if the job was cancelled.
 
local future = pool:mapAsync(data, MyWorker)

future:OnComplete(function(results, ok)
    if ok then
        print("done with", #results, "results")
    end
end)

-- elsewhere, if the result is no longer needed
future:Cancel()

When a Future carries a failure, its errors field holds the same structured error list described in Error handling.

Pool info and telemetry

pool:prewarm(module)

Loads and caches the module on every worker so the first real call does not pay the require and synchronization cost. The map family calls this on its own the first time it sees a module, so explicit prewarm is only useful to move that one-time cost out of a latency-sensitive moment.

pool:getSize() · pool:getFreeCount() · pool:isClientPool()

pool:getSize()       -- number of workers
pool:getFreeCount()  -- workers currently idle
pool:isClientPool()  -- true if the pool runs on the client

pool:getStats()

Returns a snapshot of the pool's current state and the values the cost model has learned. Cheap to call and safe to poll.

print(string.format("latency floor: %.2f ms", stats.latencyFloorMs))

for moduleName, costMs in stats.costModel do
    print(string.format("%s: %.4f ms/element", moduleName, costMs))
end

pool:destroy()

Releases the pool and everything it owns: cancels any Futures still waiting in the submit buffer, tears down the result event, and destroys all worker actors. After destroy, any call that needs a worker raises "pool is destroyed". It is idempotent; calling it twice is harmless.

A pool is meant to live across many calls. Destroy it on the lifecycle boundary that owns it (system shutdown, player leaving), not after each batch. The cost to avoid is creating and destroying pools frequently, since each creation spawns actors and runs a calibration.

Error handling

A worker can fail on a single element without taking down the rest of the batch. Each element runs under its own pcall, so one bad value produces one recorded error, not a lost chunk.

Return shape

map and mapTables return results, errors. results holds one entry per input element, in order. errors is nil when nothing failed, otherwise a list of { index, message }, where index is the position in the original input even when the work was chunked across workers.

Continue mode (default)

A failed element leaves nil in its slot in results and adds an entry to errors. The rest of the results stay intact.

local results, errors = pool:map(data, MyWorker)

if errors then
    for _, e in errors do
        warn(string.format("element %d failed: %s", e.index, e.message))
    end
end

Strict mode

The first failed element raises instead of being collected. Use it when one failure means the whole result is invalid.

local pool = ParallelFlow.createPool({ errorMode = "strict" })

local ok, results = pcall(function()
    return pool:map(data, MyWorker)
end)

onError

onError receives a descriptive string for failures, timeouts and lost chunks. It is independent of errorMode: it still fires in continue mode as a logging side channel, and in strict mode it fires before the error is raised. The structured { index, message } data lives on the errors return value, not here.

local pool = ParallelFlow.createPool({
    onError = function(message)
        warn("[pool] " .. message)
    end,
})

Futures and total failures

For async methods and submit, a failure settles the Future rather than returning an errors list; its errors field carries the same entries and Await returns nil. A timeout or worker crash is different from an element error: the whole chunk is lost, reported through onError, leaving the affected slots empty.

Supported types

Data sent to and returned from a worker crosses the actor boundary, so it is serialized into a buffer and decoded on the other side, automatically and without annotation.

Built-in

Primitives and containers: nil, boolean, number, string, buffer, arrays, and maps. Numbers are tagged integer or float internally so integer-heavy data uses a tighter encoding; arrays and maps nest freely.

Roblox types: Vector2, Vector3, CFrame, Color3, UDim, UDim2, Rect, Ray, Region3, NumberRange, BrickColor, Enum, DateTime, TweenInfo, ColorSequence, NumberSequence, and more than 25 in total, including Vector2int16, Vector3int16, Region3int16, PhysicalProperties, Axes, Faces, PathWaypoint and the curve-key types.

Custom types

A type the serializer does not know is registered once, before use. A codec declares a fixed stride in bytes and a pair of functions that write to and read from a buffer at a given offset.

local TypeCodecs = require(game.ReplicatedStorage.ParallelFlow.TypeCodecs)

TypeCodecs.register("MyType", {
    stride = 8,
    encode = function(buffer, offset, value)
        -- write `value` into `buffer` at `offset`, using `stride` bytes
    end,
    decode = function(buffer, offset)
        -- read `stride` bytes from `buffer` at `offset` and return the value
        return value
    end,
})

Once registered, values of that type can appear anywhere in the data passed to a worker, the same as a built-in.

SharedTable

Above the configured thresholds, large or nested inputs are passed through a SharedTable rather than copied per chunk. This is automatic for map, and explicit for mapInto.

Notes and limits

  • Parallelism is bounded by the engine, not by size. size sets how many worker Actors the pool creates, but Roblox decides how many run at once by scheduling Actors across a small pool of worker threads. That thread count is environment-dependent and not guaranteed: it has differed between Studio and live games and has changed over time. At the time of writing it is roughly 8 in Studio and lower in a published experience (around 3 on the client and about 2 on the server), and Roblox has indicated it may instead scale with CPU cores in the future. Creating more Actors than there are worker threads does not raise this ceiling, so a large size mainly helps in Studio and on higher-core machines; treat these figures as current behavior, not a contract.
  • A single large monolithic payload does not benefit much from chunking: the transfer cost dominates and there is little compute to spread. This is a property of the actor model, not a defect.
  • The small-input latency floor is roughly one round trip on the test machine. Below the crossover point the adaptive scheduler runs the work sequentially, so a small input is not penalized by the floor.
  • The distributed package is the runtime only. The demos and benchmark scripts live in the repository, not in the package.
